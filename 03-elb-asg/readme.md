# Scalability and High availability

Scalability means that an application / system can handle greater loads by adapting

There are two kinds of scalability
- Vertical
- Horizontal: elasticity

## Scalability
Goal: Handle more load (users, requests, data).

“Can my system grow without breaking or slowing down too much?”

Scalability = capacity growth

## High availability
Goal: Keep the system running all the time, even when failures happen.

“If something breaks, does my system stay online?”

How its achieved:
- Redundancy (multiple servers)
- Failover systems
- Replication (databases)
- Load balancers

High Availability = fault tolerance + uptime

Scalability → Auto-scaling groups
High Availability → Multi-zone / multi-region deployments

# What is load balancing?

Load balancers are servers that foward traffic to multiple servers downstream

Why use Elastic Load Balancer?

1. Classic load balancer (v1, old generation)

3. Network loadbalancer: TCP, TLS, UDP

# Sticky sessions
It is possible to implement stickines so that the same client is always redirected to the same instance
behind a load balancer
- Works for: Classic load balancer, application load balancer, and network load balancer
- The cookie used for stickines has an expiration data yor control 

## Cookie names
- Application based cookies
    - Custom cookie
        - Generated by the target
        - Can include any custom attributes requried by the application
        - Cookie name must be specified individually for each target group
        - Dont use AWSALB, AWSALBAPP, AWSALBTG 
    - Application cookie
        - Generated by the load balancer
        - cookie name is AWSALBAPP
- Duration-based cookies
    - Cookie generated by the load balancer
    - Cookie name is AWSALB for ALB,

# SSL/TLS
- An SSL certificate allows traffic between yor clients and load balancer to be encrypted in transit (in-flight encryption)

- SSL refers to Secure Sockets Layer, used to encrypt connctions
- TLS refers to Transport Layer Security, which is a newer version
- Now TLS are mainly used but people still refer as SSL

The load balancer uses an X.509 certificate (SSL/TLS server certificate)


SSL - Server name indication (SNI)

- SNI solves the problem of loading multiple SSL certificates onto one web server (to server multiple websites)
- ONly works for ALB and NLB

## ELB - SSL certificates
- Classic load balancer (v1)
    - Support only one SSL certificate
    - Must use mutiple CLB for multiple hostname with multiple SSL certificates
- Application load balancer
    - Supports multiple listeners with multiple SSL certificates
    - Uses servername indication (SNI)
- Network load balancer
    - SUpports multiple listeners with multiple SSL certificates
    - Uses SNI

# Auto scalling groups
The of an Auto Scalling Group(ASG) is to:
- Scale out(add new EC2 instances)
- Scale in (remove EC2 instances)

ASG are free (you only pay for the underlaying EC2 instances)

You need a launch template 
- AMI + instance type
- EC2 user data
- EBS volumes
- Security groups
- SSH key pair
- IAM roles for you EC2 instances
- Network + subnets info
- Loadbalancer info

It is possible to scale an ASG based on CloudWatch alarms and based on the alarm
- We can create scale-out policies
- We can create scale-in policies

# Step by stemp

1. Create A launch template
2. Create the targe group
3. Create the loadbalancer
4. Create auto scaling group
5. Attach ASC to target group

# Scaling poliices
- Target tracking scaling
    - ASG CPU to stay at around 40%
- Simple/ step scaling
    - When a CloudWatch alarm is triggered, then add 2 units
    - When a CloudWatch alarm is triggered them remove 1
- Scheduled scaling
    - Anticipate a scaling based on a known usage patters
- Predictive scaling

Metrics to scale on:
- CPU utilization
- RequestCountPerTarget
